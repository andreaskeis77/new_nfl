## 5. Bekannte Restrisiken

1. **ADR-0030 (UI Tech Stack: Jinja + Tailwind + htmx + Plot) bleibt `Proposed`.** Die Implementierung seit T2.6A ist live und stabil; der Status-Flip wurde in T2.7F nicht durchgeführt, weil T2.7 den UI-Stack nicht berührt hat. Kandidat für Re-Review im Rahmen von T3.0 wenn Lasttest-Feedback vorliegt.
2. **ADR-0032 (Bitemporale Roster-Modellierung) bleibt `Proposed`.** Die Trade-/Release-Heuristik ist in T2.5D implementiert und unit-test-validiert, aber nicht gegen echte NFL-Trades produktions-validiert. T3.0-Testphase soll dieses Gate liefern.
3. **Ruff-Baseline 45 Errors auf main.** Ruff 0.15.10 ist strenger als die Version zum letzten Clean-Commit; die 45 Errors sind Rule-Drift (UP035/UP037/E501/I001/B905/UP012/E741), keine Code-Regression. T3.0-Aufräum-Bolzen optional; Gate bleibt „Delta 0 gegenüber Baseline".
4. **Parallel-Entwicklung im geteilten Working-Tree** war in T2.7 die teuerste Kostenstelle. Fixiert durch Memory-Update und [PARALLEL_DEVELOPMENT.md](../../PARALLEL_DEVELOPMENT.md) Retro-Block: ab nächster paralleler Tranche `git worktree add` Pflicht.
5. **HTTP-Mirror für Health-Probes deferred.** T2.7A liefert CLI-Probe; HTTP-Endpoint `/health/<kind>` wartet auf echten Web-Router (frühestens T2.6I oder T2.9). Monitoring-Scripte können aktuell nur per CLI-Shell-Exit-Code abfragen.
6. **Backup als Runner-Job fehlt.** T2.7C liefert CLI `backup-snapshot`; ein `register_job_executor`-Pfad für Cron-Style-Scheduling ist nicht Teil von v1.0 — manuelles Scheduling per Windows Task Scheduler + CLI ist der T3.0-Workaround.
7. **`events_YYYYMMDD.jsonl` (T2.7B File-Destination) läuft unbegrenzt.** Keine automatische Rotation/Retention — Analog zu `trim-run-events` ist denkbar, aber kein v1.0-Requirement.
8. **`replay-domain` macht genau eine Domain pro Call.** `--all`-Modus über alle sechs Specs wäre trivial nachzurüsten bei Operator-Bedarf; aktuell manueller Shell-Loop.
